## How to Use These Reviews

### 1. Prioritise by Severity

**Critical Issues (must fix):**

- Review first
- Plan implementation immediately
- Cannot deploy to production without fixing
- Estimated effort: varies

**Warnings (should address):**

- Plan for next sprint
- Represent significant improvements
- May affect performance or security
- Can deploy with documented plan to fix

**Suggestions (nice-to-have):**

- Consider for refactoring sprints
- Improve maintainability and performance
- Can be deferred without risk
- Good for team learning and code quality

### 2. Find Action Items

Each review clearly marks actionable items with:

```markdown
## Next Steps

1. Critical: ...
2. Warning: ...
3. Suggestion: ...
```

Copy these to your project management tool (ClickUp, GitHub Issues, etc.)

### 3. Cross-Reference Code

Each finding includes:

- Specific file paths
- Line numbers
- Code examples
- Rationale for the issue

Use this to quickly locate and understand issues in the codebase.

### 4. Assign Work

Suggested workflow:

1. Review findings with team lead
2. Create tasks in ClickUp/GitHub Projects
3. Assign to developers
4. Track progress in sprints
5. Re-run reviews after implementation to verify fixes

## Taking Action

### Creating Issues

For each critical issue:

```markdown
**Title:** [CRITICAL] Fix MD5 cache keys vulnerability
**Assignee:** [Security-minded developer]
**Sprint:** Current/Next
**Priority:** P0 - Blocker
**Labels:** security, critical

**Description:**
See: docs/REVIEWS/CODE-REVIEW-2026-01-03.md

The application uses MD5 hashing for cache keys in config/cache/backend.py:45,
which is cryptographically insecure. Replace with SHA256.

**Acceptance Criteria:**

- [ ] Cache key generation uses SHA256
- [ ] All cache keys updated
- [ ] Tests pass
- [ ] No performance regression
```

### Re-running Reviews

After making changes:

1. Run the relevant review agent to verify fixes
2. Update the review documentation
3. Track improvements in project metrics

**Commands:**

```bash
# Run code review
/syntek-dev-suite:review

# Run syntax check
/syntek-dev-suite:syntax

# Run GDPR assessment
/syntek-dev-suite:gdpr

# Run logging check
/syntek-dev-suite:logging
```

### Documentation Updates

When fixes are implemented:

1. Add a "Fixed" section to the review document
2. Update this README with resolution status
3. Link to the GitHub commit/PR that fixed the issue
4. Record metrics in docs/METRICS/

## Reading Test Results

### Quick Assessment

1. **Look at summary**: Pass rate and failure count
2. **Check coverage**: Is it above target (80%)?
3. **Review failures**: Are they new or known issues?
4. **Check performance**: Are tests running fast?

### Interpreting Results

| Result                          | Status      | Action               |
| ------------------------------- | ----------- | -------------------- |
| All tests pass, coverage > 85%  | ✅ GOOD     | Merge PR             |
| All tests pass, coverage 80-85% | ⚠️ OK       | Check coverage areas |
| Tests pass but < 80% coverage   | ⚠️ CAUTION  | Add tests            |
| Some tests fail                 | ❌ STOP     | Debug failures first |
| Many tests fail                 | ❌ CRITICAL | Stop and investigate |

### Failure Investigation

When tests fail:

1. **Read the error message**: It tells you what went wrong
2. **Check the traceback**: Shows where in code the failure occurred
3. **Reproduce locally**: `pytest tests/[test_file].py::[test_name]`
4. **Review recent changes**: What code changed before the failure?
5. **Check dependencies**: Did external services change?

## Test Metrics

### Key Metrics

| Metric            | Target  | What It Means       |
| ----------------- | ------- | ------------------- |
| **Pass Rate**     | > 95%   | Most tests pass     |
| **Code Coverage** | > 80%   | Most code is tested |
| **Test Speed**    | < 5 min | Tests run quickly   |
| **Failure Rate**  | < 5%    | Few tests fail      |

### Tracking Metrics Over Time

Results should show:

- ✅ Coverage **increasing** (adding more tests)
- ✅ Pass rate **stable or improving** (fixing bugs)
- ✅ Test speed **stable or faster** (optimising tests)
- ⚠️ New failures should **decrease quickly** (fixes applied)

## CI/CD Integration

### Automated Test Runs

Tests run automatically on:

- **Commit to branch**: All tests run
- **Pull request**: Tests must pass to merge
- **Deploy to staging**: Full test suite
- **Deploy to production**: Critical test subset

### CI/CD Status

Check test status in:

- **GitHub Actions**: `.github/workflows/`
- **CI Pipeline**: Status checks on PR
- **Build Dashboard**: CI/CD provider dashboard

### Failing CI Tests

If CI tests fail:

1. **View CI logs**: See detailed error messages
2. **Run locally**: Reproduce the failure `pytest`
3. **Debug the code**: Find and fix the issue
4. **Run tests again**: Verify fix works
5. **Push changes**: CI will re-run tests

## Analysing Results

### Code Coverage Analysis

Areas to focus on:

```
✅ > 90% coverage: Well-tested code
⚠️  80-90% coverage: Good, but some gaps
❌ < 80% coverage: Needs more tests
```

**Improve coverage by**:

1. Adding unit tests for untested functions
2. Adding edge case tests
3. Adding integration tests for workflows
4. Increasing BDD scenario coverage

### Performance Analysis

Check test duration:

```
✅ < 5 minutes: Excellent
⚠️  5-10 minutes: Acceptable
❌ > 10 minutes: Too slow, optimise
```

**Optimise performance by**:

1. Running tests in parallel where possible
2. Mocking slow external dependencies
3. Using fixtures instead of setup/teardown
4. Checking for N+1 database queries

### Failure Pattern Analysis

Look for:

- **Intermittent failures**: Tests that sometimes fail
- **Related failures**: Multiple tests failing in same area
- **Environment-specific failures**: Only on certain OS/browser
- **Performance regressions**: Tests slower than before
